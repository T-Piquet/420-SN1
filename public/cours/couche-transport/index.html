Couche Transport

Rappel encapsulation - désencapsulation - routage

Transport des données

La couche de transport sert à la communication entre les applications exécutées sur les différents hôtes.

Elle comprend 2 protocoles :

  • TCP - Transmission Control Protocol
  • UDP - User Datagram Protocol

Cette couche s’occupe de s’assurer de la cohérence de toutes les communications pouvant avoir lieu en même temps sur une seule machine. Elle identifie et segmente chacune de ces communications. Elle s’assure également de la fiabilité et du contrôle de flux de la communication.

Dans le cas TCP :

  • Numérotation des segments
  • Accuser de réception
  • Retransmission des paquets si nécessaire
  • Contrôle des taux de transmission

Dans le cas UDP :

  • Fourni les fonctions de base
  • Pas de connexion entre les hôtes
  • Meilleur effort -> les informations peuvent être perdues

Protocole TCP

Le protocole TCP est un protocole de connexion, il ´établie une session. Cela permet entre autres :

  • Garantit de livraison des messages
  • Garde l’ordre des messages
  • Contrôle le flux

Protocole UDP

Le protocole UDP n’établie pas de session.

  • Les segments perdus ne sont pas renvoyés.
  • L’ordre des messages est celui de la réception.
  • Il n’y a pas de contrôle de flux.

Numéros de port (TCP et UDP)

La gestion de plusieurs communications en simultané sur une même machine est possible grâce aux ports.

Comme les adresses MAC et IP, il y a un port source et un port de destination. Le premier sur la machine locale, le second sur la machine distante, souvent un serveur.

Attributions des numéros de port

L’attribution des ports est gérée par l’IANA (InternetAssigned Numbers Authority). Les ports sont codés sur 16 bits, ils varient donc de 0 à 65 535.

Les ports de destinations sont généralement définis statiquement en fonction de l’application, alors que les ports sources sont générés dynamiquement par l’hôte créant la requête.

Groupe de ports Gamme de numéros Description
Ports réservés de 0 à 1023 - Ces numéros de port sont réservés aux services et aux applications courants ou populaires tels que les navigateurs web, les clients de messagerie électronique et les clients d’accès à distance.
Ports inscrits de 1024 à 49151 - Ces numéros de port sont affectés par l’IANA à une entité demandeuse pour être utilisés avec des processus ou des applications spécifiques- Par exemple, Cisco a enregistré le port 1812 pour son processus d’authentification du serveur RADIUS.
Ports privés et/ou dynamiques de 49152 à 65535 - Ces ports sont également connus sous le nom deports éphémères- Le système d’exploitation du client attribue généralement des numéros de port dynamique lorsqu’une connexion à un service est lancée.- Le port dynamique est ensuite utilisé pour identifier l’application cliente pendant la communication.

Vérification des ports ouvert sur un hôte

Sous windows, la commande netstat est utilisée pour lister les ports utilisés. Sur les ancienne distribution linux cette commande fonctionne aussi, mais elle a été remplacé par la commande ss (Socket Statistics).

Processus de communication TCP

Chaque processus de demande s’exécutant sur un serveur est configuré pour utiliser un numéro de port.

  • Deux services ne peuvent pas être affectés au même numéro de port d’un serveur au sein des mêmes services de la couche transport.
  • Une application de serveur active affectée à un port spécifique est considérée comme étant ouverte, ce qui signifie que la couche transport accepte et traite les segments adressés à ce port.
  • Toute demande entrante d’un client qui est adressé à l’interface de connexion correcte est acceptée et les données sont transmises à l’application de serveur.
  • De nombreux ports peuvent être ouverts simultanément sur un serveur, chacun étant destiné à une application de serveur active.

Connexion TCP - Handshake

TCP est un protocole full duplex, où chaque connexion représente deux sessions de communication à sens unique. Pour établir la connexion, l’hôte effectue une connexion en trois étapes :

  1. Elle vérifie que le périphérique de destination est bien présent sur le réseau.
  2. Elle s’assure que le périphérique de destination à un service actif et qu’il accepte les requêtes sur le numéro de port de destination que le client qui démarre la session a l’intention d’utiliser.
  3. Elle informe le périphérique de destination que le client source souhaite établir une session de communication sur ce numéro de port.

Bits de contrôle

Tous ces handshakes sont fait en utilisant les 6 bits de contrôles de l’en-tête TCP :

  • SYN - Synchroniser les numéros de séquence utilisés dans l’établissement de connexion.
  • ACK - Indicateur d’accusé de réception utilisé dans l’établissement de la connexion et la fin de la session.
  • PSH - Fonction push (Push function).
  • FIN - Utiliser dans la fin de session.
  • RST - Réinitialisation de la connexion en cas d’erreur ou de dépassement de délai.
  • URG - Champ de pointeur urgent significatif.

Processus de communication UDP

  • Le protocole UDP n’établit pas de connexion.
  • Le protocole UDP fournit un transport de données à faible surcharge car il utilise de petits en têtes de datagrammes et n’offre pas de gestion du trafic réseau.
